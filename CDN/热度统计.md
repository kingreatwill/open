**千万级 URI、分布式部署** 的 **热度系统架构图 + 处理流程**。

核心目标是：

- 冷数据默认处理，不做重计算
- 热点数据流式发现，只追热点
- 分片聚合，避免全局排序
- 热度稳定输出，驱动 `100 ~ 3000` 副本决策

**模块职责**
- **边缘节点**
  - 只做本机短窗口预聚合，不上报全量明细
  - 每分钟只上报 `TopK URI` 和超过阈值的 URI
  - 冷 URI 直接忽略，减轻中心压力
- **消息总线**
  - 通过 `hash(uri)` 分区，让同一 URI 的统计尽量落到同一分片
  - 支持横向扩容和重放
- **聚合层**
  - 维护 URI 的滑动窗口状态和 `EMA` 热度
  - 只保留“热点候选状态”，冷数据靠 `TTL` 淘汰
- **决策层**
  - 把热度映射成副本档位，不直接用连续值
  - 做容量预算和防抖
- **控制面**
  - 把“目标副本数”翻译成“在哪些节点放/撤”
  - 负责执行可靠性，不把这些逻辑塞进热度计算里

**核心流程**
- **1. 边缘预聚合**
  - 每个边缘节点按 `1 分钟` 窗口统计：
  - `req_1m`、`bytes_1m`、`uv_1m(HLL)`、`region`
  - 本机只保留 `TopK`，比如 `Top 5000`
- **2. 热点候选上报**
  - 上报满足任一条件的 URI：
  - `req > 阈值`
  - `bytes > 阈值`
  - 进入本机 `TopK`
- **3. 分片聚合**
  - 聚合器按 `hash(uri) % N` 分 shard
  - 每个 shard 合并来自不同边缘节点的数据
  - 得到该 URI 的全局近似状态
- **4. 热度更新**
  - 每个 URI 更新：
  - `req_5m`、`bytes_5m`
  - `uv_5m`
  - `region_count`
  - `ema_heat`
- **5. 候选筛选**
  - 只有 `ema_heat` 超过阈值或进入 shard 内 `TopN` 的 URI 才继续参与决策
  - 其他 URI 默认 `100` 副本
- **6. 档位决策**
  - 把热度映射到副本档位：
  - `100 / 200 / 400 / 800 / 1200 / 2000 / 3000`
  - 只允许单轮升降一档，避免震荡
- **7. 全局裁剪**
  - 如果总副本预算超标：
  - 优先保留高收益 URI
  - 压缩大对象、回源成本低对象、副本收益低对象
- **8. 下发执行**
  - 控制面把目标副本变化下发到缓存集群
  - 执行层异步扩副本/缩副本
- **9. 结果回流**
  - 回传命中率、回源率、扩散完成率
  - 下一轮决策继续使用这些反馈做修正

**推荐状态模型**
- ```text
  URIState {
    uri_hash
    req_5m
    bytes_5m
    uv_hll_5m
    region_bitmap_5m
    ema_heat
    current_tier
    target_tier
    last_promote_time
    last_demote_time
    object_size
    miss_cost
    expire_at
  }
  ```

**推荐热度计算**
- ```text
  heat =
    0.5 * log_norm(req_5m) +
    0.3 * log_norm(bytes_5m) +
    0.1 * log_norm(uv_5m) +
    0.1 * norm(region_count)
  ```
- `log_norm` 适合长尾分布：
- ```text
  log_norm(x) = min(1, log(1+x) / log(1+p99))
  ```
- 再做平滑：
- ```text
  ema_heat = 0.3 * heat_now + 0.7 * ema_heat_prev
  ```

**副本决策图**
- ```text
  ema_heat
     │
     ├─ < T1          -> 100
     ├─ T1 ~ T2       -> 200
     ├─ T2 ~ T3       -> 400
     ├─ T3 ~ T4       -> 800
     ├─ T4 ~ T5       -> 1200
     ├─ T5 ~ T6       -> 2000
     └─ >= T6         -> 3000
  ```
- 升档规则：
  - 连续 `2~3` 个周期超过上阈值才升
- 降档规则：
  - 连续 `5~6` 个周期低于下阈值才降
- 单轮最多变 `1` 档

**为什么这样设计**
- **不做全量排序**
  - 千万 URI 做全局排序成本太高
- **只追热点**
  - 真正值得多副本的 URI 只占极小部分
- **分片聚合**
  - 把状态水平拆开，天然支持扩容
- **近似统计**
  - `HLL`、`TopK`、`CMS` 能大幅节省内存和网络
- **分档而不是连续值**
  - 控制面更稳定，容量更好预算
- **防抖**
  - 避免副本数来回跳

**时间尺度建议**
- 边缘预聚合：`10s` 或 `60s`
- 聚合窗口：`5m`
- 平滑窗口：`1h EMA`
- 决策周期：`1m ~ 5m`
- 扩缩副本执行：异步、分批

**落地时最重要的 4 个阈值**
- 本机上报阈值：决定边缘上报量
- 热点候选阈值：决定聚合层保留多少 URI 状态
- 档位阈值：决定副本数分布
- 全局预算阈值：决定系统总容量上限

**一个最小可用版本**
- 边缘节点每分钟上报本机 `Top 5000 URI`
- Kafka 按 `hash(uri)` 分 `128` 个分区
- 聚合层维护 `5m req/bytes + 1h EMA`
- 副本档位固定 `100/200/400/800/1200/2000/3000`
- 默认全部 `100`
- 只有 shard 内 `Top 1%` 才进入副本提升决策
